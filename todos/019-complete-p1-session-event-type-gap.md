# SessionEventType vs AgentEvent type gap

## Problem Statement

`data-model.md` defines `SessionEventType` with 9 variants, but `agent-provider-interface.md` defines `AgentEvent` discriminated union with only 7 variants. The 2 extra types (`status_changed`, `permission_response`) exist in the DB schema but not in the TypeScript type system.

This causes:
- `event-system.md` `materialize()` table lists `status_changed` and `permission_response` as triggering materialization, but the `match("type", event)` on `AgentEvent` cannot handle these types
- No clear definition of where `status_changed` and `permission_response` events originate or how they flow through the system

## Findings

**AgentEvent (agent-provider-interface.md) - 7 variants:**
```
message | tool_use | tool_result | permission_request
| cost_update | context_update | error
```

**SessionEventType (data-model.md) - 9 variants:**
```
status_changed | message | tool_use | tool_result
| permission_request | permission_response
| error | cost_update | context_update
```

**Gap analysis:**
- `status_changed`: AgentSession emits via `emit("status")` (separate channel from `emit("event")`). SessionRunner receives it and updates `sessions.status`. Unclear if it should also be stored in `session_events`.
- `permission_response`: Generated by SessionRunner's auto_approve logic or user manual response. Not emitted by AgentSession. Stored in `session_events` by SessionRunner directly.

**Impact on materialize():**
The materialize table in event-system.md says `status_changed` updates `status, status_confidence` and `permission_response` updates `status = 'running'`. But `materialize()` takes `AgentEvent` which doesn't include these. Either:
1. These are handled in a separate code path (not materialize)
2. A broader `SessionEvent` type should wrap `AgentEvent` + internal events

## Proposed Solutions

### Option A: Introduce `SessionEvent` superset type (Recommended)

Create a `SessionEvent = AgentEvent | StatusChangedEvent | PermissionResponseEvent` that represents everything stored in `session_events`. `materialize()` takes `SessionEvent`, not `AgentEvent`.

**Pros:** Single type covers all DB events. match builder can handle all 9 variants.
**Cons:** Adds complexity. Need to distinguish "events from agent" vs "all events".
**Effort:** Medium - touch agent-provider-interface.md, event-system.md, data-model.md

### Option B: Keep AgentEvent as-is, handle internal events separately

`status_changed` is handled in the `emit("status")` listener (not materialize). `permission_response` is handled inline in auto_approve. Remove both from the materialize table. Clarify in data-model.md that `SessionEventType` is a superset of `AgentEvent["type"]`.

**Pros:** Minimal change. Clear separation between agent-emitted and system-generated events.
**Cons:** Two code paths for session_events INSERT. materialize table is incomplete.
**Effort:** Low - mainly documentation updates

### Option C: Remove `status_changed` from session_events entirely

Status changes are already materialized in `sessions.status`. Storing them in `session_events` is redundant. Keep only `permission_response` as an internal event type.

**Pros:** Removes redundancy. Simpler event model.
**Cons:** Loses audit trail for status transitions. Timeline can't show status changes.
**Effort:** Low

## Recommended Action

**Option A: Introduce `SessionEvent` superset type.**

Rationale:
1. `session_events` is an audit ledger — all state transitions belong there, including status changes and permission responses
2. Timeline (dual-mode-ui.md) displays `status_changed` entries — they must exist in session_events
3. A single `materialize(SessionEvent)` with match builder over 9 variants is consistent with the "no switch, exhaustive match" principle
4. `AgentEvent` stays untouched in agent-provider-interface.md — it represents what agents emit. `SessionEvent` lives in event-system.md — it represents what the system records

**Changes required:**

1. **event-system.md**: Define `SessionEvent = AgentEvent | StatusChangedEvent | PermissionResponseEvent`
   - `StatusChangedEvent`: `{ type: "status_changed", source, confidence, payload: { status: SessionStatus, previous?: SessionStatus } }`
   - `PermissionResponseEvent`: `{ type: "permission_response", source: "user" | "auto", confidence: "high", payload: { requestId: RequestId, approved: boolean } }`
2. **event-system.md**: `materialize()` signature: `SessionEvent` instead of `AgentEvent`. Add 2 new match cases
3. **event-system.md**: `maybeNotify()` signature: `SessionEvent`. No-op for the 2 new types
4. **event-system.md**: `handleEvent()` and `handleStatus()` both feed into the same ledger append + materialize pipeline
5. **data-model.md**: Add note that `SessionEventType = AgentEvent["type"] | "status_changed" | "permission_response"`
6. **agent-provider-interface.md**: No changes. `AgentEvent` remains the agent-emitted contract

## Acceptance Criteria
